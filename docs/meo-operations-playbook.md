# MEO代行 運用プレイブック（受注前に固める版）

> **目的**：MEO自動化ツールの受注が入る前に、「あなた一人しか回せない」「店舗が増えると破綻する」「GoogleにアカBANされる」「レポートで詰められて解約される」の4大事故を、仕組みで潰しておく。
>
> **前提**：ツール本体の完成度に関わらず使える。運用の型が先にあれば、受注が来ても慌てない。ツールが自動でやる部分と人がやる部分を、この文書で明確に切り分ける。

---

## 0. なぜ「受注前」に運用を固めるのか

MEO代行で潰れる会社は、ツールの性能ではなく **運用（オペレーション）で潰れる**。

- 3店舗までは気合いで回る。**問題は5店舗を超えてから**一気に噴出する。
- そのとき慌てて手順を作っても遅い。**すでに納品遅延・返信漏れ・数字未計測が起きている**。
- だから「受注が入りそうで不安」は、正しい不安。**今が唯一の仕込みどき**。

このプレイブックは、受注ゼロの今のうちに以下を確定させる：

1. 属人化を消すSOP（誰がやっても同じ品質）
2. 店舗が増えても破綻しないスケール設計
3. GoogleにBANされない安全ライン（ここが一番怖い）
4. 数字で語れるレポートと、解約されない仕組み

---

## 1. サービス範囲の確定（何を売って何を売らないか）

**受注前にこれを決めていないと、店舗ごとに作業がブレて破綻する。**

### 1-1. 標準プラン（これ以外は受けない、を先に決める）

| 項目 | 標準に含む | オプション | 対象外（受けない） |
|---|---|---|---|
| GBP最適化（初期） | ○ カテゴリ・営業時間・属性・説明文 | — | — |
| 投稿代行 | ○ 週1〜2本 | 週3本以上 | — |
| 口コミ返信 | ○ 全件（自動下書き＋人が承認） | 24時間以内保証 | **サクラ口コミ生成（絶対NG／2章参照）** |
| 順位計測 | ○ 主要KW 5〜10語／週次 | 日次計測 | — |
| 写真追加 | ○ 月◯枚（店舗提供 or 生成） | 撮影代行 | 他店・ネットからの無断転載 |
| レポート | ○ 月次 | 週次 | — |

> **原則**：メニューを増やすほど属人化する。**最初は「1プラン固定」**で回し、型ができてからオプションを解禁する。

### 1-2. 成果保証はしない（明文化する）

- Googleの順位は**Googleのアルゴリズム次第**で、代行側が保証できるものではない。
- 契約書・提案書に「順位・集客数の保証はしない」と明記。**保証を匂わせた瞬間、解約時にトラブルになる**。
- 代わりに保証するのは「**作業の実施**（投稿本数・返信率・計測レポート）」。ここは100%守る。

---

## 2. Googleポリシー安全ライン（最重要・ここでBANされると全店舗巻き添え）

> **これを破ると、あなたの管理アカウント経由で契約店舗が一斉に処分される可能性がある。自動化ツールを作るなら、この章の禁止事項を"物理的にできない"設計にすること。**

### 2-1. 絶対禁止（一発アウト級）

| やってはいけないこと | 理由 |
|---|---|
| **自作自演・サクラの口コミ投稿／購入** | Googleの禁止コンテンツ。発覚で口コミ全削除・リスティング停止 |
| **報酬と引き換えの口コミ依頼**（割引・プレゼントの見返り） | ガイドライン違反（レビュー・ゲーティング含む） |
| **キーワードを店名（ビジネス名）に詰め込む** | 正式名称以外を入れると停止対象。実店舗の登記/看板名のみ |
| **住所・カテゴリの偽装、実体のない店舗登録** | 即停止＋再登録困難 |
| **他店の口コミへの妨害・虚偽通報の乱用** | アカウント単位で悪質判定 |
| **1アカウントからの大量自動操作（bot的挙動）** | スパム判定でアカBAN |

### 2-2. ツール設計への落とし込み（自動化する人向け）

- 口コミは**「返信の下書き自動生成」までに留め、投稿は人が承認**。口コミ本文の自動生成・自動投稿はしない。
- 投稿の自動化は**「下書き生成＋予約」まで。** 1日の投稿回数・API叩き頻度に上限を設け、bot的な連打をしない。
- 複数店舗を1つのGoogleアカウントで管理する場合、**ビジネスグループ（旧ロケーショングループ）の正規オーナー権限**で運用。裏技的な一括操作ツールは使わない。
- ネガティブ口コミへの対応は「**削除申請できるのはガイドライン違反のもののみ**」。気に入らないだけの低評価は消せない。返信で対応する型を用意。

### 2-3. グレーゾーンの線引き（判断に迷ったら安全側）

- ○ 実際の来店客に「よかったらGoogleに感想を」と**フラットにお願いする**（見返りなし・星数を指定しない）
- ✕ 「★5でお願いします」「レビューくれたら10%OFF」→ NG
- ○ 口コミ集めのQR/リンクを店内に置く（全員に均等に見せる）
- ✕ 高評価しそうな客だけにリンクを見せる（レビュー・ゲーティング）→ NG

## 3. 属人化を消すSOP（あなた以外でも回る状態にする）

> **不安の正体の多くはこれ。「自分が倒れたら全店止まる」状態を消す。**

### 3-1. 店舗オンボーディング手順（受注直後にやること）

1. **ヒアリングシート回収**（店名・登記名・カテゴリ希望・営業時間・強み・NGワード・既存GBPのオーナー権限）
2. **GBP権限の受領**（管理者 or オーナー権限。ここで詰まると何も始まらない）
3. **初期監査**：現状のカテゴリ・写真・口コミ・NAP（名称/住所/電話）整合性・競合3社をチェック → 監査メモを残す
4. **対策KWの確定**（主要5〜10語。地域＋業種＋ニーズ）
5. **初月コンテンツカレンダー作成**（投稿ネタ4本／写真◯枚）
6. **計測ベースライン記録**（開始時点の順位。これがないと後で成果を語れない）

### 3-2. 週次ルーティン（1店舗あたり）

| 曜日 | 作業 | 自動/手動 |
|---|---|---|
| 月 | 順位計測・前週差分チェック | 自動（ツール）→ 人が確認 |
| 火 | 投稿1〜2本の下書き生成→承認→予約 | 半自動 |
| 随時 | 新規口コミに返信（下書き自動→人が承認、目標24〜48h以内） | 半自動 |
| 金 | 週次の異常チェック（順位急落・口コミ炎上・情報改変） | 自動アラート＋人 |

### 3-3. チェックリスト方式で品質を揃える

- 作業は**「記憶」でなく「チェックリスト」でやる**。抜けが事故になる領域（口コミ返信漏れ・投稿予約ミス）は特に。
- テンプレ化するもの：投稿文トーン、口コミ返信の型（★5用／低評価用／クレーム用）、レポート雛形。
- **属人化の指標**：「自分がいなくても回るか？」を毎月自問。回らないなら、その作業を文書化orツール化する。

## 4. 複数店舗のスケール設計（ここで破綻する）

### 4-1. キャパシティの現実値（受注ペースの目安）

- **半自動運用なら、1人あたり10〜20店舗が現実的な上限**（投稿・口コミ・レポートの承認作業がボトルネック）。
- それを超えるなら、**(a) 自動化率を上げる／(b) 人を増やす／(c) 受注を絞る** の三択。無理に受けると全店の品質が落ちる。
- **受注は「同時に何店舗まで面倒見られるか」を先に決めてから取る**。埋まったら待ってもらう or 単価を上げる。

### 4-2. 店舗が増えても破綻しない仕組み

| 仕組み | 内容 |
|---|---|
| 台帳の一元管理 | 全店の「ステータス・次アクション・計測値・契約更新日」を1枚のシート/DBで管理（下記4-3） |
| バッチ処理 | 順位計測・投稿下書きは店舗横断で一括生成。1店ずつ手作業にしない |
| 例外だけ人が見る | 正常な店は自動で流し、**異常（順位急落・低評価・情報改変）だけアラートで人に上げる** |
| テンプレの共有 | 業種別の投稿・返信テンプレを持ち、新店はそこから複製して微調整 |

### 4-3. 運用台帳（最小カラム案）

このリポジトリの `data/` に沿って、店舗運用台帳を持つと良い。最小カラム：

```
店舗ID, 店名, 業種, 契約開始日, 契約更新日, プラン, GBP権限状態,
対策KW, 開始時順位, 最新順位, 今月の投稿数, 未返信口コミ数,
次回レポート日, ステータス(正常/要対応/解約リスク), 担当, メモ
```

> `ステータス` を毎週1回だけ全店ぶん更新すれば、「どこがヤバいか」が一目で分かる。これが**スケール時の生命線**。

## 5. レポートと解約防止（3ヶ月で切られないために）

> **「で、順位上がったの？」に数字で即答できないと切られる。** MEOは効果が出るまで時間がかかるので、**"作業してます"を可視化し続ける**ことが継続のカギ。

### 5-1. 月次レポートに必ず入れる4要素

1. **順位の推移**（対策KWごと。開始時 → 今月。上下どちらも正直に）
2. **実施した作業**（投稿◯本／口コミ返信◯件・返信率◯%／写真◯枚）← 保証しているのはココ
3. **数字の変化**（GBPのインサイト：表示回数・電話・ルート検索・サイト誘導）
4. **来月やること**（次アクション。放置してない感を出す）

### 5-2. 解約を防ぐ運用習慣

- **効果が出るまで時間がかかることを、契約時に握っておく**（最低3ヶ月は見てほしい、と先に伝える）。あとで言うと言い訳に聞こえる。
- 順位が動かない月でも**「作業実績」と「インサイトの微増」で継続の納得感**を作る。沈黙が一番切られる。
- 低評価・炎上は**即エスカレーション**。放置が信頼を最も削る。
- 契約更新日の**30日前にアラート**を出し、更新面談で「次の一手」を提案する（受け身で待たない）。

## 6. 受注前チェックリスト（このプレイブックの実行版）

受注が入る前に、以下が「Yes」になっているか：

- [ ] 標準プランを1つに固定した（メニューを増やしすぎていない）
- [ ] 成果保証はしないと契約書/提案書に明記した
- [ ] Google禁止事項（2章）を、ツール設計上"できない"ようにした
- [ ] 口コミ・投稿は「自動下書き＋人が承認」の型になっている
- [ ] オンボーディング手順書（3-1）を1枚にまとめた
- [ ] 投稿文・口コミ返信のテンプレを業種別に用意した
- [ ] 同時に受けられる店舗数の上限を決めた（例：まず◯店まで）
- [ ] 運用台帳（4-3）を作り、ステータス列を毎週更新する運用にした
- [ ] 月次レポートの雛形（5-1）を作った
- [ ] 契約時に「効果は最低3ヶ月」を握る営業トークを用意した

---

## 7. 次の一手（このドキュメントの使い方）

1. **1〜2章を最優先で確定**（プラン固定・BAN対策）。ここが事故ると全部飛ぶ。
2. **3・4章を"実物"に落とす**：オンボーディング手順書・運用台帳・テンプレ集を作る。必要なら `data/` にテンプレJSON/CSVを置く。
3. **5章のレポート雛形を1枚作る**。受注1店舗目でそのまま使う。
4. ツール自動化は、この運用フローの「人が承認する半自動」を壊さない範囲で進める。

> このプレイブックは叩き台。実際のツール仕様・プラン内容が固まったら、ここに追記して"生きた運用マニュアル"に育てる。
